# 卡牌游戏开发踩坑记录

> 从零搭一个 Godot 4.6 卡牌游戏过程中犯过的所有错误，分门别类整理。
> 每个坑都标了：**现象 → 原因 → 教训**。

---

## 一、项目组织（全局性错误）

### 1. 一把梭，全崩盘

**现象**：一次性写了 20+ 个文件，打开 Godot 报了 30+ 个错误，一个都跑不起来。

**原因**：想一口气搭完"完美架构"，EventBus / 状态机 / 效果系统全上。但实际上每一步都有语法错误和理解偏差，叠加起来根本没法排查。

**教训**：**先跑起来，再完善。** 哪怕脚本只有 10 行，能让一张卡牌显示在屏幕上，都比 500 行但全是红字强。每次只加一个功能，加完就测试。

> 后来被要求"全部删干净，一步一步做"，才恢复正常进度。

---

## 二、Godot 语法/API 错误

### 2. `mouse_filter = 2` 点不了卡牌

**现象**：卡牌显示正常，但怎么点都没反应。

**原因**：`mouse_filter` 的枚举值是：
- `0` = `MOUSE_FILTER_PASS`（穿透）
- `1` = `MOUSE_FILTER_STOP`（拦截）
- **`2` = `MOUSE_FILTER_IGNORE`（完全忽略鼠标）**

我写了 `mouse_filter = 2`，等于告诉 Godot"这张卡无视鼠标"。

**教训**：**记住枚举值。** `STOP = 1`，`IGNORE = 2`。不熟的 API 先查文档，不要猜。

### 3. `mini()` 在 Godot 4 已废弃

**现象**：编译警告，运行时可能出错。

**原因**：Godot 3 用 `mini(a, b)` 取整数最小值，Godot 4 统一用 `min()` 处理所有数值类型。

**教训**：**从 Godot 3 迁移时，`mini → min`。**

### 4. `Tween.stop_all()` 不存在

**现象**：运行时报错 "Nonexistent function 'stop_all' in base 'Tween'"

**原因**：Godot 4 的 `Tween` 没有 `stop_all()` 方法。正确做法是 `tween.kill()` 然后建新的。

**教训**：**Godot 4 的 Tween API 和 Godot 3 不一样。不要以为方法名相似就能用。**

### 5. Signal 参数名用了 `max`

**现象**：所有 signal 行报错。

```gdscript
signal player_energy_changed(current, max)  # ← 错误
```

**原因**：`max` 是 GDScript 内置函数，不能当参数名用。

**教训**：**`min`、`max`、`name`、`type` 这种关键词/内置函数名，不要用作变量名或信号参数名。**

### 6. `name` 作为局部变量

**现象**：编译警告 "local variable 'name' is shadowing an already-declared property in the base class 'Node'"

```gdscript
var name = card_data.card_name  # ← 冲突！Node 基类已经有 name 属性
```

**教训**：**Node 基类的属性不要覆盖。常见的冲突：`name`、`position`、`rotation`、`scale`。**

### 7. 子场景节点没加 `parent="."`

**现象**：主场景加载后布局错乱，子节点出现在错误位置。

**原因**：在 `.tscn` 文件中，子节点必须写 `parent="."`，否则 Godot 把它当成第二个根节点。

```gdscript
# 错误（Hand 成了独立根节点）
[node name="Hand" type="Node2D"]

# 正确
[node name="Hand" type="Node2D" parent="."]
```

**教训**：**手写 tscn 文件时，子节点必须显式指定 parent。**

### 8. `preload` 放在文件作用域

**现象**：卡牌场景有问题时，整个主脚本都无法加载（连 `_ready` 都不跑）。

```gdscript
const CardScene = preload("res://scenes/card.tscn")  # ← 编译期加载
```

**原因**：Godot 4 的 `const preload()` 是在脚本编译期执行的。如果目标文件有任何问题，脚本直接编译失败，不会进入运行阶段。

**教训**：**能用 `load()` 在 `_ready` 里运行时加载，就不要用 `const preload` 在编译期加载。** 运行时加载有错误起码能捕获，编译期加载直接不运行。

## 三、场景/节点设计错误

### 9. 卡牌场景引用不存在的子节点

**现象**：卡牌实例化时报错。

```gdscript
@onready var sprite: Sprite2D = $Sprite2D  # 场景里没有这个节点
```

**原因**：重构卡牌画面时，把 Sprite2D 换成了 ColorRect，但忘记更新 `@onready` 引用。

**教训**：**改场景结构时同步改脚本引用，或者统一用 `$NodeName` 不声明 @onready 变量。**

### 10. 布局全用绝对坐标，窗口缩放就消失

**现象**：缩小窗口后卡牌被裁掉甚至完全看不见。

**原因**：场景里的节点位置写死了 `Vector2(640, 500)`，没有做任何自适应缩放。

**教训**：**在 `project.godot` 里加 stretch 模式：**

```
[display]
window/stretch/mode = "canvas_items"
window/stretch/aspect = "keep"
```

这样内容会等比缩放，不会因为窗口缩小而跑出画面。

## 四、类型标注错误

### 11. 给 @onready 变量加了 `: Node` 类型标注

**现象**：脚本编译正常，但 `_ready` 不执行，所有功能失效。

```gdscript
@onready var player_node: Node = $PlayerPanel/Player
# 然后下一行：
player_node.hand_node = hand  # ← Node 类型没有 hand_node 属性
```

**原因**：Godot 4 的类型系统会在编译时检查「声明的类型上有这个属性吗」。`Node` 没有 `hand_node` 属性，GDScript 给出 **编译错误**（不是警告）。

**教训**：**鸭子类型有时比强类型更稳定。如果变量在运行时一定是 Player 类型，直接 `@onready var player_node = $Player` 不加类型标注，或者用 `Player` 类型：**

```gdscript
@onready var player_node: Player = $Player  # 正确！Player 有 hand_node
```

**关键：标注的类型必须真的包含你要访问的所有属性和方法。**

### 12. 内类在泛型数组里不稳定

**现象**：`Array[StatusData]` 时灵时不灵。

```gdscript
class StatusData:  # Player 的内类
	var effect_type: int
var statuses: Array[StatusData] = []  # ← 不稳定
```

**教训**：**内类不建议用在泛型数组里。要么拆成独立文件（有自己的 class_name），要么用 Dictionary 代替。**

### 13. `owner` 属性被覆盖

**现象**：敌人 AI 引用自己的节点时行为异常。

```gdscript
var owner: Node  # ← 覆盖了 Node 基类的 .owner 属性！
```

**教训**：**`owner` 是 Node 的保留属性（指向场景所有者），不要用它做变量名。换成 `enemy_node` 或 `my_node`。**

## 五、逻辑/设计错误

### 14. z_index 倒置

**现象**：最早加入手牌的卡遮住了最新加入的。

```gdscript
card.z_index = i          # 第1张 z=0，第2张 z=1……第7张 z=6
						   # 结果：最早的最靠上，最新被挡住
```

**正确**：最新的卡应该在视觉最上层。

```gdscript
card.z_index = count - i  # 第7张 z=7，第1张 z=1
```

### 15. 切牌时旧卡滞留空中

**现象**：选中卡 A → 点卡 B → 卡 A 浮在空中不下落。

**原因**：`_select_card` 只设置了选中卡的 y 坐标，没把上一张选中卡复位。

**教训**：**状态切换时要考虑"退出旧状态"的逻辑，不能只设新状态。** 每次都存 `prev` 变量，对上一张选中卡单独做复位动画。

### 16. `_gui_input` 没消费事件

**现象**：点击卡牌时浮起 0.1 秒后立刻落下（"颤动"）。

**原因**：卡片的 `_gui_input` 处理了点击，但没有调用 `set_input_as_handled()`。Godot 认为这个事件没被消费，又触发了 `_unhandled_input` → `hand.deselect()`，导致选中和取消选中同时发生。

**教训**：**Control 节点处理完输入事件后，调用 `get_viewport().set_input_as_handled()` 消费事件，防止 `_unhandled_input` 重复触发。**

### 17. 切牌后透明度没恢复

**现象**：先选卡 A（其他卡变暗），再选卡 B → 卡 B 也变暗。

**原因**：`_select_card` 只把新选中卡的 `position:y` 设为浮起，没把它的 `modulate` 恢复成全亮度。

**教训**：**状态的每个维度都需要完整迁移。只改位置不改透明度就是半残状态迁移。**

### 18. Hand 没有 max_hand_size 检查

**现象**：`max_hand_size` 声明了但 `add_card` 里没判断。

```gdscript
var max_hand_size: int = 10  # 声明了
func add_card(card) -> bool:
	cards.append(card)        # 直接加，不检查上限
```

**教训**：**声明了限制条件，就要在入口处强制校验。否则等于没有。**

### 19. 敌人/玩家死亡后游戏继续

**现象**：敌人 HP 归零后还能继续出牌；玩家被敌人打死动画照常播放。

**原因**：处理伤害的逻辑里只改了 HP 数值，没有检查是否死亡，也没有终止后续流程。

**教训**：**每次修改 HP 后（无论是受伤还是治疗），都应该触发死亡检查。用一个 `game_over` 标志位阻止后续操作。**

### 20. 能量双倍回复

**现象**：每个回合结束时能量从 3 变成 6（或从 0 变成 6）。

**原因**：`game_manager.gd` 的 `start_enemy_turn` 里已经调用了 `player.refill_energy()`，`main.gd` 的 `_on_end_turn` 又调了一次。

**教训**：**一个操作只在一个地方做。谁负责补能量要想清楚——要么 game_manager 全管，要么 main.gd 全管，不要交叉。**

### 21. 引用不存在的场景节点

**现象**：`@onready var deck_label = $DeckLabel`，但场景里没有 `DeckLabel` 节点。

**原因**：在 main.gd 加了新 UI 的引用，但忘记更新 main.tscn 加上对应节点。

**教训**：**加新功能时 checklist：1) 在 tscn 加节点 → 2) 在 gd 引用 → 3) 连接信号。顺序不能乱。**

## 六、整体设计教训总结

| 问题 | 一句话教训 |
|------|-----------|
| 一次做太多 | **先跑通再完善，每一步都要能测试** |
| 不熟 API 直接猜 | **不查文档就等着踩坑**（Tween、mouse_filter） |
| 类型标注乱加 | **鸭子类型比错误类型强更稳定** |
| 忽略状态迁移 | **退出旧状态和进入新状态同样重要** |
| 忘了加节点 | **先在场景放节点，再在代码引用它** |
| 死亡没判定 | **每次改 HP 后必须检查生死** |
| 信号事件不消费 | **`set_input_as_handled()` 是必须的** |
| 双重复位 | **同一件事只在一个人手下干** |
