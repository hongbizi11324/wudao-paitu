## 一、杀戮尖塔风格地图系统

### 1.1 整体架构

```
GameData (持久层)           node_map.gd (UI层)          main.gd (控制层)
	│                            │                           │
	├─ generate_new_act()        ├─ open()                   ├─ _show_map()
	├─ select_map_node()         ├─ _draw_lines()            ├─ _on_node_selected()
	├─ map_layers[]              ├─ _on_node_clicked()       └─ ..._closed()
	├─ map_connections[]         ├─ _on_node_hover()
	└─ map_node_states[][]       └─ _build_ui()
```

**数据与UI分离原则**：GameData 只存数据结构和算法逻辑，node_map.gd 只负责渲染和交互。main.gd 做流程编排。

### 1.2 算法：大关切分

**问题**：武道牌途的楼层是无限递增的（每6层一个Boss），怎么切成杀戮尖塔的"大关"？

**方案**：从战斗起始层+1开始，到下一个Boss层结束。

```
第1大关 = {楼层: [2, 3, 4, 5, 6]}  （Boss在第6层）
  起点：打完第1层战斗后进入
第2大关 = {楼层: [8, 9, 10, 11, 12]}（Boss在第12层）
  起点：打完第6层Boss后进入
```

**公式**：
```
start_floor = map_act_count * 6 + 2
map_act_count: -1 → 0 → 1 → 2 ...
初始化 -1 的原因：第一次 generate_new_act() 会 ++ 到 0，start_floor = 2 ✓
```

**为什么跳过第7层？** 因为杀戮尖塔的设计是打完Boss直接进入下一个大关，中间没有过渡战斗。武道牌途也沿用这个设计。

---

### 1.3 算法：路径连接

**目标**：每层节点之间有随机连线，保证从起点到Boss至少有一条可达路径。

**算法分两步：**

**Step A — 随机连接（每个节点主动出线）：**
```
for 每层 L 的每个节点 N:
	targets = 随机取下层 1-2 个节点
	for 每个 target T:
		connections.append({from: (L, N), to: (L+1, T)})
```

**Step B — 覆盖修补（保证下层无孤岛）：**
```
for 下层 L+1 的每个节点 T:
	if T 没有入线:
		source = 上层 L 的随机节点
		connections.append({from: (L, source), to: (L+1, T)})
```

**为什么需要 Step B？** 纯随机连接时，可能出现下层某个节点没有任何节点连向它，导致该节点永远不可达。

**时间复杂度**：O(N*M)，其中N=层数（固定5），M=每层节点数（固定3），可视为常数。

### 1.4 算法：节点类型分配

**约束**：
- 普通层：至少1个战斗节点
- 精英层：战斗节点池中加入精英战斗
- Boss层：固定1个Boss节点（居中）
- 避免同一层3个节点全是战斗（遍历检查，超了换非战斗）

**分配池**：
```
普通层池子 = [战斗, 商店, 休息, 事件]
精英层池子 = [战斗, ⚔精英战斗, 商店, 休息, 事件]
Boss层 = [♛Boss]
```

**分配逻辑**：
```
1. 先放1个战斗节点（确保最少1个）
2. 剩余位置从池子里随机选
3. 如果战斗节点已≥2个，强制换成非战斗
4. 最终打乱顺序
```

### 1.5 算法：状态迁移机

每个节点有3种状态，状态转换如下：

```
				   选中该节点
  LOCKED 🔒 ──────────────→ AVAILABLE ✅ ──────→ VISITED ✓
  不可见/不可点                 可选高亮               已走过变灰
	  ↑                            │                       │
	  │                  点击解锁同层其他节点锁住            │
	  │                            │                       │
	  └──── 未连接任何上层节点 ─────┘                       │
														   │
											 选完后解锁下层可达节点
```

**触发时机**：
- `select_map_node(layer, idx)` 被调用时触发状态变迁
- 选中节点 → VISITED，同层其他 → LOCKED，下层可达节点 → AVAILABLE

**关键保证**：一旦选中一个节点，同层其他节点立即锁死，防止玩家在同一层选多个节点。

## 二、Bug 修复记录

### Bug 1: `_draw()` 覆盖 CanvasItem 内置回调

**发现时间**：2026-06-28，编写 node_map.gd 时

**现象**：
```gdscript
func _draw():  # ← 这是 CanvasItem._draw() 的回写！
	var line = Line2D.new()
	...
	canvas.add_child(line)  # ← 在 _draw() 里创建子节点
```

**原因**：`_draw()` 是 Godot CanvasItem 的内置虚方法，每帧或调 `queue_redraw()` 时自动调用。我在里面创建 Line2D 子节点而不是用 `draw_line()` 图元，导致：
1. 每次重绘都创建新的 Line2D，旧的永远不会被清理
2. 逐渐积累数不清的 Line2D 节点，内存泄漏

**修复**：改名为 `_draw_lines()`，在 `open()` 末尾手动调用。用 `_clear_map()` 每次打开时先清理 `_draw_control` 的所有子节点。

```gdscript
# 正确做法
func _clear_map():
	for c in _panel.get_children():
		c.queue_free()
	for c in _draw_control.get_children():  # ← 新增！清理旧连线
		c.queue_free()

func _draw_lines():
	# 创建 Line2D 子节点（不是用 _draw() 回调）
	var line = Line2D.new()
	...
	_draw_control.add_child(line)
```

**教训**：**不要覆盖你不知道的 Godot 内置方法。** `_draw`、`_process`、`_input`、`_ready` 都是引擎回调，如果你要自定义函数，取个不同的名字。

---

### Bug 2: 连线/发光环从未清理（内存泄漏）

**发现时间**：与 Bug 1 同时发现

**现象**：每次 `open()` 地图，旧的连线和发光环 ColorRect 都留在场景树里，新的又加上去。

**原因**：`_clear_map()` 只清理了 `_panel` 下的子节点，没清理 `_draw_control` 下的子节点。

**修复**：在 `_clear_map()` 末尾加：
```gdscript
for c in _draw_control.get_children():
	c.queue_free()
```

**教训**：**创建子节点的函数和清理子节点的函数必须对称。** 在哪里 `add_child()`，就要在哪里 `queue_free()`。

---

### Bug 3: `_on_draw()` 空函数

**发现时间**：2026-06-28

**现象**：
```gdscript
func _on_draw():
	call_deferred("_draw")
```

**原因**：设计时想用 `draw.connect("draw", _on_draw)` 但没连接，留下一个死函数。且 `call_deferred("_draw")` 试图调用被覆盖的 `_draw()`，造成无限递归风险。

**修复**：完全删除。

**教训**：**不要写不连接的信号处理函数。** 如果你写了但没有 `connect`，它永远不会执行，只会迷惑阅读者。

---

### Bug 4: `map_act_count` 初始值错误

**发现时间**：2026-06-28，流程调试时

**现象**：`generate_new_act()` 第一次调用后，`start_floor = 1 * 6 + 2 = 8`，但第一关应该从第2层开始。

**原因**：
```gdscript
var map_act_count: int = 0
# generate_new_act() 里:
map_act_count += 1  # 0 → 1
start_floor = 1 * 6 + 2  # = 8 ❌
```

**修复**：初始值改为 -1，第一次 increment 后变为 0，`0 * 6 + 2 = 2` ✓

```gdscript
var map_act_count: int = -1
```

**教训**：**边界初始值要仔细算。** 自增型计数器的初始值 = 想要的第一次值 - 1。

---

### Bug 5: `_on_event_closed` 的 advance_floor 没被替换

**发现时间**：2026-06-28，修改 main.gd 时

**现象**：替换 `advance_floor()` 为 `_show_map()` 时，因为 `_on_event_closed` 后面跟的商店段的 `advance_floor()` 连在一起，Edit 匹配时只匹配了商店段，漏掉了事件段。

**原因**：edit 工具的 oldText 匹配了 `\nGameData.advance_floor()\n\tnode_map.open()\n\n\n# 商店`，但 `_on_event_closed` 末尾也有一段相同的 `\nGameData.advance_floor()\n\tnode_map.open()`，两段相邻，匹配只覆盖了后面那个。

**修复**：手动检查每个 `advance_floor` 的引用，逐一替换。

**教训**：**全局替换后要 grep 确认没有残余。** 批量替换时宁可多匹配几次。

## 三、Godot 性能优化笔记

### 3.1 子节点创建 vs `_draw()` 图元

创建节点的开销远大于 `draw_line()`。但如果节点数量很少（本系统：最多30条线 + 15个节点），用子节点更简单直观，不影响性能。

**法则**：
- 静态/低频创建（每关一次） → 用子节点，代码简单
- 高频更新（每帧变） → 必须用 `draw_*()` 图元

### 3.2 `queue_free()` 的正确姿势

`queue_free()` 不会立即删除节点，而是在当前帧结束后统一清理。所以：

```gdscript
# 安全——先标记删除，再创建新的不会冲突
for c in old_children:
	c.queue_free()
for i in range(new_count):
	var c = new_child()
	add_child(c)
```

**不能**在同一个节点上既 `queue_free()` 又立即使用它的引用。

### 3.3 StyleBoxFlat vs 实时绘制

每次创建 StyleBoxFlat 对象有微小开销。本系统每次 open() 重建约15个 StyleBoxFlat，在1280x720下完全可接受。

如果以后需要频繁刷新（每帧换颜色），应该考虑缓存 StyleBox 对象或改用 `modulate`。

### 3.4 悬停效果的性能

悬停时改变 `scale` 会触发 Godot 的 CanvasItem 重绘。当前仅场景中15个节点可能被悬停，性能无影响。

如果需要大量节点（>100），悬停放缩必须用 `draw_set_transform()` + `draw_rect()` 替代 `scale`。

## 四、MCP 工具架构

> 项目根目录的 `mcp-server.js` 是一个独立 Node.js 服务，通过 Model Context Protocol 暴露以下工具：

| 工具名 | 功能 | 实现方式 |
|--------|------|---------|
| `project_info` | 项目概况 | 解析 project.godot + 文件数统计 |
| `list_scripts` | 脚本列表 + 结构分析 | 解析 GDScript AST（正则提取 class_name/extends/func/signal） |
| `read_script` | 读文件内容 | 直接 fs.readFile |
| `list_cards` | 卡牌列表 | 解析 .tres 资源的键值对 |
| `read_card` | 卡牌详情 | 同上 |
| `search_code` | 代码搜索 | 逐行 grep |
| `card_balance_report` | 平衡性分析 | 遍历卡牌数据计算费用/伤害比等 |
| `dependency_graph` | 类依赖 | 扫描 extends/class_name/preload/load |
| `error_database` | 踩坑记录查询 | 解析 Markdown 章节 |
| `scene_structure` | 场景节点树 | 解析 .tscn 的 [node] 行 |

**JSON-RPC 2.0**：MCP 协议基于 JSON-RPC 2.0，通过 stdio 交换请求/响应。每个请求是 `{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{...}}`。

## 局域网联网关键教训

1. 操作同步（operation-based）会逐渐漂移，状态快照（state-based）直接覆盖更可靠
2. Godot @rpc 的 authority / any_peer / call_local 组合要谨慎，选错就是异步根源
3. autoload 加载顺序影响编译时标识符可见性
4. 手牌同步不能用全量重建（闪烁），必须 diff 增量更新
5. 写日志！print 出牌/快照，两边对比才能找到不同步点
